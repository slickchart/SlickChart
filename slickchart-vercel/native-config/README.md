# native-config — staging for native app config files

These files belong in the **native app projects** (generated by `cap add ios` / `cap add android`),
NOT in the web app or on the server. They're staged here so they're version-controlled and already on
your Mac after you clone/pull.

## After you run the cap:add commands, copy them into place:

**Android** — after `npm run cap:add:android`:
```bash
cp native-config/android/google-services.json android/app/google-services.json
```

**iOS** — after `npm run cap:add:ios`, add `GoogleService-Info.plist` to the iOS app **in Xcode**
(drag it into the `App` target — see FIREBASE-SETUP-BABYSTEPS.md Part C). When you have that file, drop
it at `native-config/ios/GoogleService-Info.plist` here too and I'll keep it with the set.

Then run `npm run cap:sync`.

## What's here
- `android/google-services.json` — Firebase Android config. Project **slickchart-2097d**, package
  **com.slickchart.app**. (Safe to commit — this file is designed to ship inside the Android app; its
  API key is restricted to your package. It is NOT the service-account secret.)
- `ios/GoogleService-Info.plist` — Firebase iOS config. Same project, bundle **com.slickchart.app**.
  (Also safe to commit — it's designed to ship inside the iOS app, and is NOT the service-account
  secret.) On the Mac it belongs in the iOS app target — added via Xcode at `ios/App/App/`.

## What is NOT here (and must never be committed / pasted in chat)
- The **service account key** (Firebase → Project settings → Service accounts → Generate new private
  key). That one is a real secret — paste it directly into **Vercel** as the `FIREBASE_SERVICE_ACCOUNT`
  environment variable. It's what lets the server SEND push.
